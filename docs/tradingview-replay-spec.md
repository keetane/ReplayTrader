# 1分足リプレイ Web アプリ 仕様書

作成日: 2026-07-05

## 1. 目的

ユーザーが自分で入手した銘柄別 1 分足 CSV をブラウザ上で読み込み、TradingView のリプレイに近い操作感で過去相場を再生・検証できる Web アプリを作る。

初期版は「過去データの閲覧・リプレイ・仮想売買」に限定する。実注文、投資助言、売買推奨、シグナル配信、自動売買は実装しない。

公開は GitHub Pages を想定し、CSV はサーバーへアップロードしない。すべての CSV パース、リプレイ、仮想売買、セッション保存はユーザーのブラウザ内で完結させる。

## 2. 前提データ

確認済みの CSV 形式:

```csv
Datetime,Close,High,Low,Open,Volume
2026-02-19 09:05:00+0900,4145.0,4170.0,4130.0,4170.0,0
```

開発者の動作確認用データディレクトリ:

```text
/Users/keetane/Documents/trade/trader/1min
```

ファイル名例:

- `141A.csv`
- `1579.csv`
- `^N225.csv`

基本仕様:

- ユーザーが選択した CSV ファイル名を銘柄 ID として扱う。
- `Datetime` は `+0900` 付きの JST として解釈する。
- OHLC は数値、`Volume` は数値として解釈する。
- 欠損分足は存在しないバーとして扱い、初期版では自動補完しない。
- 取引所の昼休み・寄前・大引け後などのセッション判定は、データに存在するバーを正とする。
- ユーザー CSV はサーバーに送信しない。
- ブラウザの File API で読み込み、必要に応じて IndexedDB にローカル保存する。

## 3. スコープ

### MVP

- CSV ファイル選択
- 複数 CSV 読み込み
- 読み込んだファイルから銘柄選択
- 1 分足ローソク足チャート表示
- 出来高表示
- 日付指定。指定日がない場合は最も近い日付へフォールバック
- 1分足/5分足の切り替え
- MA5/MA25/MA60の表示。選択日の先頭では前日以前のデータも使う
- リプレイ開始位置の指定
- 再生、一時停止、1 本送り、巻き戻し
- 再生速度変更
- リプレイ中の仮想成行・指値注文
- 仮想ポジション、平均建値、評価損益、確定損益の表示
- 売買履歴の保存
- リプレイセッションの保存・再開
- ダークモードをデフォルト表示し、ライト/ダークを切り替える
- 初期資金をデフォルト500万円とし、画面から変更できる
- 信用維持率を表示する
- 日跨ぎ建玉がある場合は新規注文を拒否する

### MVP ではやらない

- 実注文送信
- 投資助言、売買シグナル配信、自動売買
- CSV のサーバーアップロード
- 管理者提供の実在相場データ配信
- 有料サービス化
- 手数料、金利、貸株料、逆日歩、税金の計算
- 日跨ぎ建玉の保有
- TradingView の有料/非公開 Charting Library への依存
- 外部マーケットデータの再配布

## 4. 推奨技術方針

### フロントエンド

- React + TypeScript
- Vite などの静的ビルド構成
- チャートは `lightweight-charts` を第一候補にする。
- UI は左に銘柄/リプレイ設定、中央にチャート、右に注文/ポジション/履歴を配置する。

理由:

- 1 分足の大量バー表示に強い。
- TradingView 製の OSS ライブラリで、Apache 2.0 ライセンスと明記されている。
- Advanced Charts/Trading Platform よりライセンス条件が軽く、ローカル検証アプリに向いている。

### バックエンド

- MVP ではバックエンドを持たない。
- GitHub Pages でホストできる静的 Web アプリにする。
- CSV 読み込み、銘柄一覧、バー管理、セッション保存はブラウザ内で実行する。
- 開発者の動作確認でテストデータを同梱/参照する場合は、一般公開前にパスワード保護された検証環境だけで使う。

### 永続化

MVP:

- リプレイセッション、仮想注文、約定、ポジションを IndexedDB に保存する。
- 明示的なエクスポート/インポート機能を用意する。
- CSV 原本を IndexedDB に保存するかどうかはユーザー選択にする。

将来:

- 複数端末同期を行う場合は別設計にする。
- 同期機能を入れる場合でも、CSV や取引ログをサーバー保存する前にデータ利用規約とプライバシー方針を再確認する。

## 5. アーキテクチャ

```text
Browser File Input
  -> Client Data Loader
  -> In-memory/IndexedDB Bar Store
  -> Replay Engine
  -> Chart UI
  -> Paper Trading Engine
  -> IndexedDB Session Store
```

### 主要モジュール

`DataLoader`

- CSV ファイルを読み込む。
- カラム検証、型変換、時刻ソート、重複除去を行う。
- パース失敗行をエラーとしてレポートする。
- ブラウザ内で完結し、ファイル内容をネットワーク送信しない。

`SymbolRepository`

- CSV ファイル名から銘柄一覧を返す。
- 銘柄名メタデータは初期版では持たない。

`BarRepository`

- 指定銘柄・期間の OHLCV を返す。
- メモリ上のバー配列または IndexedDB から取得する。

`ReplayEngine`

- 現在の再生インデックスを管理する。
- `play`, `pause`, `stepForward`, `stepBack`, `seek`, `setSpeed` を提供する。
- チャートへ「現在時点までのバー」を供給する。

`PaperTradingEngine`

- リプレイ時点の価格だけを使って仮想約定を作る。
- 成行は現在バーの `Close` または次バー `Open` のどちらで約定させるかを設定可能にする。
- 指値はバーの `High/Low` 到達で約定判定する。
- 手数料、金利、貸株料、逆日歩、税金は初期版では計算しない。

`OrderGateway`

- MVP では仮想注文のみ。
- 実注文は実装しない。
- UI やリプレイエンジンから証券 API を呼ばない。

## 6. 画面仕様

### メイン画面

左ペイン:

- CSV ファイル選択
- 読み込み済みファイル一覧
- 銘柄検索
- 日付範囲
- リプレイ開始日時
- 表示足種別: 1分 / 5分
- 日付指定: 未指定時は今日に最も近い取引日を表示。指定日にデータがない場合は最も近い取引日を表示。

中央:

- ローソク足チャート
- 出来高
- クロスヘア
- 現在リプレイ時刻ライン
- 注文/約定マーカー

下部コントロール:

- 再生/一時停止
- 1 本戻る
- 1 本進む
- 速度: `1x`, `5x`, `10x`, `30x`, `60x`
- 現在時刻
- 表示中バー数

右ペイン:

- 現在値
- 仮想資産
- 初期資金入力。デフォルトは500万円。
- 信用維持率
- 建玉一覧
- 注文フォーム
- 注文一覧
- 約定履歴
- 損益サマリー

### 注文フォーム MVP

項目:

- 売買: 買い / 売り
- 取引区分: 現物 / 信用新規 / 信用返済
- 注文種別: 成行 / 指値
- 数量
- 指値価格
- 有効期限: 当日相当。日跨ぎ建玉は不可。

MVP では信用新規/返済も仮想売買として処理する。実注文 API には接続しない。信用返済で返済可能数量を超える注文は拒否し、返済注文が新規建玉に転じないようにする。

## 7. 信用取引トレーニングの設計方針

実注文には接続しないが、信用取引のトレーニングとして以下のドメインモデルを持つ。

`Account`

- 現金
- 初期資金
- 建玉評価額
- 維持率
- 実現損益

`Position`

- 銘柄
- 建区分: 信用買建 / 信用売建
- 数量
- 平均建値
- 建日
- 評価損益

`Order`

- side: buy / sell
- tradeType: cash / marginOpen / marginClose
- marginSide: long / short
- orderType: market / limit / stop
- quantity
- price
- status

`RiskCheck`

- 注文前に余力、保証金率、空売り可否、建玉上限を確認する。
- この結果は仮想チェックとして表示だけに使う。
- 実際の証券会社の信用余力、銘柄別規制、増担保、逆日歩等を保証しない。
- 初期版の信用維持率は `口座評価額 / 建玉評価額 * 100` の簡易計算とする。
- 手数料、金利、貸株料、逆日歩、税金は計算対象外とする。

## 8. 法的・規約リスク調査メモ

これは実装判断用の整理であり、法律意見ではない。実注文、自動売買、売買推奨、サーバーへの CSV 保存、第三者への相場データ配信を始める前に弁護士または金融規制に詳しい専門家へ確認する。

### 8.1 GitHub Pages 公開時の基本リスク

GitHub Pages で静的アプリだけを公開し、ユーザー CSV をブラウザ内で処理する方式なら、アプリ運営者が相場データを取得・保存・再配布しないため、データ再配布リスクを大きく下げられる。

確認事項:

- データ入手元が個人利用・分析利用を許しているか。
- CSV をアプリに読み込むことが「再配布」「二次利用」「商用利用」に該当しないか。
- 取得元が証券会社、情報ベンダー、スクレイピング先の場合、保存・加工・再利用の制限がないか。
- 利用規約で、ユーザーが自分の責任で CSV の利用権限を確認することを明記する。

### 8.2 データ再配布リスク

マーケットデータは、取引所・情報ベンダー・配信元の契約で再配布が制限されることが多い。アプリを外部公開し、CSV の内容や派生データを第三者へ見せる場合は、データ再配布に該当する可能性がある。

設計上の対応:

- MVP は GitHub Pages の静的アプリにする。
- CSV は `<input type="file">` で読み込み、サーバーへ送信しない。
- ユーザー自身が各自のデータを読み込む形にする。
- サンプルデータを同梱する場合は、架空データまたは明確に配布可能なデータだけにする。
- 開発者が入手した実在相場 CSV は公開ビルドに含めない。
- 開発者の動作確認環境で実在テストデータを使う場合は、パスワード保護し、一般公開しない。

### 8.3 投資助言・金融商品取引業リスク

単なるチャート表示、過去検証、ユーザー自身の仮想注文記録にとどまる限り、ただちに投資助言とは言いにくい。

一方、次の機能はリスクが上がる。

- 特定銘柄の売買推奨
- 自動売買シグナル
- ユーザーごとの資産状況に応じた助言
- 成果報酬や月額課金での投資判断提供
- 第三者資金の運用
- 実注文の自動発注

金融庁は、無登録で金融商品取引業を行う者への警告情報を公開している。外部提供や助言性のある機能を入れる場合は、金融商品取引業・投資助言・代理業への該当性を確認する。

実装上の禁止ライン:

- MVP では「買うべき」「売るべき」などの推奨文言を出さない。
- ランキングやシグナルを出す場合も「検証用の機械的条件」と明示し、個別推奨に見えない UX にする。
- 他人の口座で注文しない。

### 8.4 実注文・信用取引リスク

実注文機能は実装しない。実注文機能を入れると、単なる可視化ツールから取引執行システムに性質が変わるため、別プロジェクトとして再検討する。

主なリスク:

- 証券会社 API 利用規約違反
- 誤発注
- 注文状態の不整合
- 信用余力・保証金維持率・空売り規制・銘柄別規制の見落とし
- 自動売買/助言/媒介に該当する可能性

実装上の対応:

- 証券 API キーや口座情報を入力する UI を作らない。
- 実注文モードを作らない。
- 注文フォームは「仮想注文」「トレーニング用」と明示する。

### 8.5 TradingView 関連リスク

「TradingView のリプレイを実装」と表現しているが、実装では TradingView の商標、画面、独自機能をコピーしたサービスとして見えないようにする。

確認済み事項:

- TradingView Advanced Charts は、公開環境かつ TradingView 表示維持などの条件がある。
- Advanced Charts/Trading Platform は自前データソースを接続する前提で、ライブラリ自体にマーケットデータは含まれない。
- TradingView の Terms は、別途許可がない商用利用や TradingView コンテンツ/データの再配布に制限を置いている。
- Lightweight Charts は TradingView 製の OSS で、Apache 2.0 ライセンスと案内されている。

設計上の対応:

- アプリ名や UI 文言に `TradingView` を使わない。
- 「TradingView 互換」「TradingView クローン」として公開しない。
- 初期実装は `lightweight-charts` を使い、独自 UI とする。
- TradingView 由来のデータ、スクリーンショット、Pine Script、インジケーター実装を無断流用しない。

### 8.6 著作権・データベース権的な注意

日本法上、単純な数値データ自体の著作物性は限定的でも、データベース、選択・体系化、契約上の利用制限は別問題になる。データ提供元の契約に従うことを優先する。

設計上の対応:

- データの出所メタデータを保存できるようにする。
- 将来の外部公開時は、配布可能なデータだけを対象にする。
- ユーザーアップロード型にする場合も、利用者に権利確認を求める。

### 8.7 個人情報・秘密情報

ローカル利用では低リスクだが、実注文連携では口座情報、API キー、注文履歴、資産情報を扱う。

対応:

- API キーをブラウザに保存しない。
- `.env` と OS キーチェーンを使う。
- ログに API キー、取引パスワード、口座番号を出さない。
- 実注文ログは改ざんしにくい形式で保存する。
